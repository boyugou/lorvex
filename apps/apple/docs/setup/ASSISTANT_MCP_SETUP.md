# Connecting Your AI Assistant to Lorvex

Lorvex uses [MCP (Model Context Protocol)](https://modelcontextprotocol.io) over **stdio** to connect with AI assistants. Your assistant launches the Lorvex MCP host locally — no daemon, no network port.

This is the best automation experience available. Lorvex remains a strong standalone app even when MCP is unavailable or not practical on a given device.

## Quickest Setup

**Option A: From the App**

1. Open **Lorvex** → **Settings** → **Assistant**
2. Click **"Copy Setup Prompt"**
3. Paste it into your AI assistant (Claude, Codex, etc.)
4. The AI will configure itself automatically

**Option B: Manual Config**

In **Settings → Assistant**, expand the **Advanced** disclosure to reveal the manual MCP client config, then use **Copy Config** (or **Copy Command Path**) and paste the snippet into your specific client.

## Manual Setup

### Where Client Config Files Live (macOS)

| Client | Config File Location |
|--------|---------------------|
| Claude Desktop | `~/Library/Application Support/Claude/claude_desktop_config.json` |
| Claude Code | `~/.claude.json` |
| Codex | `~/.codex/config.toml` |

### Config Format Examples

**Claude Desktop** (JSON — add to `mcpServers` object):
```json
{
  "mcpServers": {
    "lorvex": {
      "command": "<path from Lorvex Settings>",
      "args": ["<args from Lorvex Settings>"]
    }
  }
}
```

**Claude Code** (JSON — add to `mcpServers` object):
```json
{
  "mcpServers": {
    "lorvex": {
      "type": "stdio",
      "command": "<path from Lorvex Settings>",
      "args": ["<args from Lorvex Settings>"]
    }
  }
}
```

**Codex** (TOML — add `[mcp_servers.lorvex]` section):
```toml
[mcp_servers.lorvex]
command = "<path from Lorvex Settings>"
args = ["<args from Lorvex Settings>"]
startup_timeout_sec = 20
tool_timeout_sec = 120
```

The actual binary path and arguments are generated by Settings → Assistant and are specific to your installation.

### Verification

After configuring, run one read tool to verify:
```
get_overview
```
It should return your task lists and stats.

## For Developers (Source Checkout)

Build the MCP host binary:

```bash
# Build all Swift targets (includes LorvexMCPHost)
swift build -c release

# The binary will be at:
.build/release/LorvexMCPHost
```

Point your MCP config at the absolute path of the built binary:
```json
{
  "mcpServers": {
    "lorvex": {
      "type": "stdio",
      "command": "/path/to/lorvex/.build/release/LorvexMCPHost",
      "args": []
    }
  }
}
```

Every Lorvex surface — the app, widgets, App Intents, notifications, and the
MCP helper — opens the single Lorvex-managed App Group store; there is no
storage selection, and cross-device sync is CloudKit-only. Settings → Assistant
never emits a database override, so the pasted config carries no `env`.

`LORVEX_APPLE_DB_PATH` is for an **unsandboxed dev/source build** only — one run
directly from `swift build`, or packaged ad-hoc via `script/package_local.sh`
(its `dist/Lorvex.app` output is ad-hoc signed, not sandboxed). Source-checkout
developers may set `LORVEX_APPLE_DB_PATH` for an explicit fixture database; when
unset, the MCP host runs the real Swift core against Lorvex-managed local
storage. A sandboxed/packaged build (Mac App Store, notarized Developer ID, or
anything under `/Applications`) serves the managed store only. There is no
user-facing environment switch for preview/in-memory mode.

## Tool Calling Input Shape

Prefer canonical JSON types:

- Numeric fields as JSON numbers: `priority: 1`, `target_count: 3`
- Arrays as JSON arrays: `tags: ["work", "urgent"]`
- Dates as ISO strings: `due_date: "2026-03-15"` (also accepts `today` / `tomorrow`)
- Boolean fields must be JSON booleans, numeric fields must be JSON numbers

## MCP & AI Integration

### Write Retry and Idempotency

Retryable write tools expose an optional `idempotency_key`. Generate a fresh opaque key for each intended mutation. If the assistant loses its session, times out, or sees a transient failure after sending the request, retry the same tool call with the exact same payload and the same key.

Lorvex caches successful write responses by `(tool_name, idempotency_key)` for approximately 24 hours and replays the cached response for matching retries.

If a host stops after the mutation commits but before its full response is stored, the retry returns `idempotency_response_unavailable` instead of running the mutation again. Read the affected records to recover current state; do not retry the same intent under a new key.

**Do not reuse an `idempotency_key` for an edited payload or a different tool call.** The server compares a `request_checksum` of the payload against the stored checksum and rejects mismatched reuse. This prevents stale cached responses from masking silent data divergence — a reused key with a changed payload is always an error, never silently replayed.

### Prompt-Injection Fencing

User-supplied string content (task titles, notes, list names, memory entry content) in MCP read responses is wrapped in `⟦user⟧…⟦/user⟧` sentinels (U+27E6 / U+27E7). These sentinels mark the boundary between system-structured content and user-controlled text so AI clients can distinguish user input from system instructions within the same JSON payload.

AI clients should treat text between these sentinels as user content and apply appropriate trust levels. Lorvex-controlled fields (IDs, status enums, timestamps, counts) are never fenced.

### Local-trust model (helper access)

The MCP helper runs locally and speaks MCP over stdio to whatever client you configure to launch it. There is no
client-side authentication token: **any local process that can launch the helper has the same access to your Lorvex data
(read, export, modify) as the assistant you configured.** This is the standard MCP-over-stdio trust model — the helper
trusts its launching client, and you control which client that is via your MCP configuration.

The effective threat is therefore local malware already running on your machine, which would already have direct access
to your Lorvex database file (and your other files) regardless of the MCP helper. A per-connection token would not change
this: the token would live in the same client-config file that a local attacker can read. So Lorvex deliberately does not
add one — it would add friction without a real security boundary. Keep your MCP client configuration private, as you would
any local credential.

### ai_changelog — Audit Trail

Every MCP mutation passes through the canonical `ai_changelog` funnel. Unless the user has explicitly selected the `"off"` retention policy, the mutation transaction records a human-readable summary, the tool name, and a before/after state snapshot. This log is visible in the app (Settings → Diagnostics → AI Changelog) and readable via the `get_ai_changelog` MCP tool. Full-content entries sync across the user's devices and converge by immutable audit id; preview and dry-run entries stay on the originating device. Retention is user-configurable through the preference API key `ai_changelog_retention_policy` (`"maximum"` to retain up to 10,000 entries, a positive integer number of days, or `"off"` to stop recording and clear existing entries on every device). The key is a virtual API surface backed by dedicated, account-scoped CloudKit retention metadata; it is never emitted as an ordinary synced preference row. Each account carries a monotonic retention frontier that rejects retired generations and out-of-window records. Before any retained audit record can be uploaded, the app records exact account-and-zone cloud-presence evidence; pruning removes local full-content copies and queues durable physical CloudKit record deletion for the exact account and zone where presence was possible. No audit tombstone or sync-delete envelope is used. Candidate-zone snapshots include every retained audit row and are sealed only after their retention witness remains stable. Cleanup stays durable while iCloud sync is disabled and resumes when sync is enabled later.

## Environment Variables

`LORVEX_APPLE_DB_PATH` is honored only by an **unsandboxed dev/source build**
(run directly from `swift build`, or packaged ad-hoc via
`script/package_local.sh`). A sandboxed/packaged build (Mac App Store,
notarized Developer ID, or anything under `/Applications`) always opens the
Lorvex-managed App Group store — the single store every surface shares.

- **`LORVEX_APPLE_DB_PATH`** — Optional path to a Lorvex SQLite fixture database, for a dev/source build only. When unset, the MCP host opens Lorvex's default managed database.

## Known Client Differences

- **Claude Desktop** — Works well with absolute binary paths in the `command` field
- **Claude Code** — Requires `"type": "stdio"` in the config; prefer absolute binary path
- **Codex** — Supports `startup_timeout_sec` and `tool_timeout_sec` fields; use these for slower machines
- After any MCP config change, restart the assistant process/session to apply changes
- Only one active Lorvex MCP registration per client config at a time
